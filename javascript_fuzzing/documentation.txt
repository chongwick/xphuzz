January 16:
I'm using Fuzzilli's coverage calculation to calculate the coverage of each JS engine execution. I modified JavaScriptRaider's Python wrapper. This Python wrapper is used to access and interpret the bitmaps genrated by the Fuzzilli coverage.
The first step is instrumenting a JS Engine to enable fuzzing. Blank JS code is fed to this engine 150 times. There is some uncertainty involved with fuzzing. By running these "dummy executions," we can get an initial bitmap of the engine which will be used as the "base map" from which to measure subsequent executions. Beyond 150 executions, there isn't any increase in coverage for the base map. Additionally, this base map of the JS engine is saved and loaded during each fuzzing session. This prevents the need to reperform the base map generation. It is worth noting that with updates or changes to the JavaScript engine, the base map must be updated.
Initially, the fuzzer takes a corpus directory as input. For each seed in the corpus, the fuzzer creates a bitmap. This collection of bitmaps is stored in a directory. 
Next, all possible combinations of 2 are generated from the input corpus. A random combination of base and ancillary seed are selected to be intermingled as the first mutation.
This summarizes the pre-fuzzing process.

A combination is fed to the LLM (i.e. "Here is Code A... Here is Code B..."). These are mixed together. Argument parsing added.

January 18:
Some seeds don't produce any coverage on top of the base calculation. One of two things can be done here. 1) We assume that the base calculation is too high and that the seeds are actually valid. We disregard the base calculation in this case. 2) We remove the 0-coverage seed from the corpus. This assumes something is wrong with the seed.
